Consent Management System

A multi-tenant solution for managing customer consent across sites — covering opt-in/opt-out tracking, expiration policies, audit trails, and CSV reporting.

Core concepts

🗂

Data points

Foundational building blocks representing specific types of data that can be collected or processed. Loaded from classpath JSON at startup.

📋

Categories

Group related data points under a single consent topic. Customers grant or revoke consent at the category level.

✅

Consent choices

Records a customer's opt-in or opt-out decision for a specific category, with optional expiry support.

⏱

Expiration

Per-category expiry duration with daily scheduler. Email reminders sent N days before expiry with de-duplication.

How it works

  1. Tenant setupEnable consent management via POST /customer/consent with { "enabled": true }. On service startup, ConsentTenantBootstrap loads system categories and data points from JSON. File changes are tracked via SHA-256 hashes for idempotent updates.
  2. Category configurationView data points via GET /customer/consent/datapoints, create custom categories via POST /customer/consent/categories, then configure which are active via POST /customer/consent/configurations. Each change is versioned.
  3. Recording consentCall POST /customer/profiles/{customerId}/consents with { categoryId, consentStatus }. The system validates the category, stores the choice, creates an audit entry, and writes a YYYYMM-partitioned history row.
  4. Expiration handlingIf configured, consents get an expiresAt timestamp. The ConsentExpiryScheduler runs daily to mark expired records. The ConsentExpiryReminderNotifier sends email reminders N days before expiry.
  5. Policy notificationsWhen policies change, trigger POST /customer/consent/configurations/notify to email all affected customers.

Features

Consent toggle

Enable or disable consent management at the tenant level. All consent operations are locked when disabled.

Custom categories (CRUD)

Full create, read, update, patch, and delete — both single and batch. System categories and opted-in categories are protected from deletion.

Configuration versioning

Every configuration change is versioned. Full history available via GET /customer/consent/configurations/history.

Consent expiration

Global toggle per tenant. Per-category duration settings. Option to retroactively apply expiry to existing consents.

Email notifications

Automated expiry reminder emails (de-duplicated, configurable days-before). On-demand policy update notifications.

Audit trail

Every consent change is recorded with full audit metadata. Category lifecycle events (create, update, delete) are all tracked.

CSV reporting

Query consent history by tenant, customer, category, and date range. Download as CSV via POST /customer/consent/history/download.

Tenant bootstrap

System categories and data points auto-loaded from classpath JSON on startup. File hash tracking prevents duplicate loads.

Authorization & permissions

Service-level roles bypass individual permission checks. All other consent endpoints require role-based authorization.

Authentication schemes

API KeyX-Upstart-Api-KeyAPI key based authorization for service-to-service calls

Bearer JWTAuthorization: Bearer ...Required for customer-facing endpoints (e.g. recording consent choices)

Permission levels

customer.consent.read

View categories, configurations, and choices. No prerequisites.

customer.consent.create

Create categories and configurations. Requires read.

customer.consent.update

Update categories, configurations, and expiry. Requires read + create.

customer.consent.delete

Delete categories and configurations. Requires read + create + update.

customer.consent.report.download

Download consent history CSV. Requires read.

Partitioning strategy

Customer choices

Partitioned by (tenant, customer_id, category_id) for fast per-customer lookups.

History reports

Partitioned by (tenant, year_month) with clustering on (created_at, category_id, customer_id, event_id) for efficient date-range queries.

Expiry lookups

Partitioned by (tenant, expiry_date) with clustering on (category_id, customer_id) for daily batch processing.

Configuration

# application.conf
customer {
  consent-expiration-scheduler {
    intervalHours = 24  # how often the expiry scheduler runs
    lookbackDays  = 7   # days to look back for missed expirations
  }
  consent-expiry-notifier {
    runAtHour   = 8   # hour (MT) to send reminders
    runAtMinutes= 0   # minutes past the hour
    beforeDays  = 7   # days before expiry to send reminder
  }
}
